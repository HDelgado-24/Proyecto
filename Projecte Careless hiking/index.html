1️. Introducción

        1.1. ¿Qué se realizará en este proyecto?

En este proyecto se realizará el desarrollo de un videojuego 2D de plataformas, estilo Pixel Art, orientado en un mundo fantasioso y medieval y un paisaje montañoso, con el uso del software de Unity.

Debido al escaso tiempo del que se dispone, se realizará una versión demo del juego que abarque la jugabilidad, hasta la primera batalla del enemigo.

1.1.1. ¿Qué historia cuenta nuestro juego?

La historia recita a un campesino que un día decide ir a un pueblo cercano a una gran montaña. En el pueblo, nuestro protagonista se encuentra con la familia real: el rey y la princesa. Nuestro campesino, al ver a la princesa, se enamora, y empieza a perseguirla por el pueblo, provocando así que la princesa se asuste y huya de él.

Los guardias de la princesa, al ver la situación, deciden escoltarla hasta lo alto de la montaña para intentar ponerla a salvo, pero nuestro protagonista, al pensar que está siendo secuestrada, se aventura a subir la montaña para rescatar a la princesa.

1.2. ¿Por qué este proyecto?

Se ha decidido hacer este proyecto más ambicioso debido a la gran afición a la industria de los videojuegos que tienen los autores del proyecto.

Además, ambos autores tienen la intención de continuar su formación en grado superior de Desarrollo de Aplicaciones Web. Por lo tanto, se vio este proyecto como una buena oportunidad de hacer uso de entornos de programación, como se han enseñado en los módulos de Aplicaciones Ofimáticas, Aplicaciones Web e Iniciación a la Programación. Esto contribuirá a otorgar una base de cara a los posteriores años.

2️. Descripción del reto

2.1. ¿Qué habilidades se aprenderán durante este proyecto?

        

En este proyecto se aprenderá a hacer uso del motor gráfico de Unity.

Los autores aprenderán a realizar un grupo conjunto de usuarios de Unity para poder compartir la carpeta del juego mediante su desarrollo.

Los autores tendrán que realizar versiones cada vez más completas del juego, a la vez que parches para arreglar los errores que se puedan encontrar por el camino.

También se realizará scripts en lenguaje de programación C#.

Se desarrollará una mejor comprensión y conocimiento de la sintaxis de la programación al desarrollar este proyecto. Cabe destacar que los autores prevén continuar con este proyecto para llegar a realizar una versión completa del juego.

Por último, se adquirirán buenas prácticas tras este proyecto.

Se harán copias de las distintas versiones del juego cuando se vea necesario, tanto en la nube mediante los servicios de Unity, como en local con un Pendrive. Este segundo caso es bastante más lento, pero se busca tener una segunda opción de recuperar los datos del juego en caso de una pérdida repentina, o de no poder acceder a los servicios de Unity.

        2.2. ¿Qué se requerirá para este proyecto?

Para este proyecto se requerirá cuatro equipos: dos equipos en el centro educativo para progresar con el proyecto en las horas asignadas en el horario escolar, y uno en el hogar de cada autor para poder progresar con el proyecto fuera del horario escolar.

Se necesitará la última versión de la aplicación Unity Hub.

Esta aplicación será la principal herramienta para el desarrollo del proyecto. Cuenta con tutoriales y plantillas para realizar videojuegos, tanto en 2 dimensiones como en 3 dimensiones.

También se utilizará un software externo para realizar nuestros propios sprites. Un sprite es una imagen o animación bidimensional que se integra en un juego.

En este caso, los autores optaron por Aseprite, un software de pago que contiene muchas herramientas para facilitar la creación de sprites.

Por último, se hará uso de un Pendrive para realizar las copias de seguridad del proyecto.

4️. Desarrollo técnico

        4.1. ¿Cómo se realizó el proyecto?

Para iniciar, los autores del proyecto, una vez decidido el proyecto, comenzaron con crear un documento con lluvia de ideas sencillas, para guiar cómo realizar el proyecto. En la lluvia de ideas se decidió que el género del videojuego sería un plataformero y su contexto histórico sería en la Edad Media.

Una vez decidido el estilo del videojuego, se comenzó creando bocetos a mano sobre papel. Ahí se decidió el aspecto del protagonista y del primer enemigo al que se encontrará.

Continuó la decisión de motor de diseño de videojuegos. Existen tres motores principales: Unreal Engine, Unity y GoDot. Para realizar videojuegos 2D las opciones más sencillas son Unity y GoDot, siendo esta última la más sencilla, pero también la más reciente, teniendo menos tutoriales online para guiarse. Es por todo ello que se decidió utilizar el motor de Unity.

Los autores instalaron Unity Hub, la aplicación/plataforma principal de Unity. Unity Hub cuenta con distintas plantillas para videojuegos y almacenamiento en la nube para proyectos personales, asociados a la cuenta en la que se inició sesión.

Se usó la plantilla para videojuegos 2D (Core 2D). Se creó un grupo de usuarios para compartir el proyecto entre los dos usuarios.

Se hizo uso de assets gratuitos de la Unity Asset Store para elementos básicos y elementos temporales que se usaron de plantilla. Los assets son los distintos elementos gráficos de un videojuego, puede variar entre assets animados o estáticos.

Con los tiles (bloque de mapa/terreno) de un conjunto de assets, se hizo el terreno principal visualmente, y posteriormente se le añadieron sus respectivos hitboxes. Los hitboxes son elementos invisibles que otorgan tangibilidad a los assets. Se le pueden añadir físicas o hacerlas inmovibles. Un claro ejemplo sería hacer un hitbox al suelo para que sea tangible y hacer una hitbox al personaje para que pueda estar encima del suelo.

4.2. ¿Qué problemas surgieron?

El primer problema que surgió fue el desconocimiento de cómo guardar los cambios del proyecto en la nube y no solo en local, para poder trabajar en conjunto desde distintos equipos.

Se encontró la manera de generar versiones del juego, pudiendo guardar cambios y retroceder a versiones anteriores en caso de errores.

El siguiente gran problema que surgió fue la escasez de sprites que se deseaban en el juego por parte de la Asset Store. Los autores, como ya se mencionó anteriormente, solucionaron este problema optando por la compra de una licencia de uso de un software especializado en la creación de sprites.

El último problema que apareció fue la necesidad de adaptarlo a un entorno web para la demostración del proyecto, y que las personas puedan probar a jugar el mismo. Unity no era capaz de interpretar el código para todo tipo de navegadores web, pero a través de una extensión del mismo llamado WebGL se pudo solucionar.

6️. Resultados y conclusiones

Se pudo completar alrededor de un 75% de la demo que se propuso en un principio, pero se consiguió realizar una demo que muestre al jugador la mecánica básica del videojuego.

Al final no se logró implementar un sistema de inventario, pero dado a su complejidad, la falta de tiempo y a que realmente no podría tener una implementación útil en la resumida demostración, decidimos excluirlo.

Se ha aprendido a realizar scripts en lenguaje de programación C#,  implementando el conocimiento básico que se obtuvo de programación en el curso en un caso práctico como es el desarrollo de un videojuego.
